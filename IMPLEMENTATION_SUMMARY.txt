HOW THE INTEGRATION WORKS:
================================================================================

Data Flow:
┌─────────────────────────────────────────────────────────────────────────────┐
│ 1. BACKEND (Off-Chain)                                                      │
│    - User completes SDG challenge                                           │
│    - Backend stores in user_challenges table                                │
│    - Creates ImpactRecord JSON with fields from schema.sql                  │
│                                                                             │
│ 2. MODULE 1 (Hash Engine)                                                   │
│    - Receives ImpactRecord dict                                             │
│    - Canonicalizes: sort keys alphabetically, compact JSON                  │
│    - Hashes with SHA-256 → 64-char hex string                              │
│    - DETERMINISTIC: Same input always gives same output                    │
│                                                                             │
│ 3. MODULE 2 (Blockchain Anchor) ← DEPLOYMENT STEP                          │
│    - Takes hash from Module 1                                               │
│    - Calls contract.anchorHash(hash_as_bytes32)                            │
│    - Emits HashAnchored event on Sepolia                                   │
│    - Returns: tx_hash, block_number, timestamp                             │
│    - Off-chain storage: saves tx_id with impact record                     │
│                                                                             │
│ 4. MODULE 3 (Verification)                                                  │
│    - Input: ImpactRecord + tx_hash (or direct hash)                        │
│    - Mode A: Fetch on-chain hash from tx_hash                              │
│    - Mode B: Use provided hash directly (offline mode)                     │
│    - Calls Module 1's verify_hash() for comparison                         │
│    - Returns: VERIFIED, TAMPERED, or MISSING                               │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

Record Schema Alignment (from schema.sql):
- user_id ..................... Pseudonymous user ID
- challenge_id, challenge_number, challenge_title ... Challenge metadata
- sdg_id, sdg_number, target_id, indicator_id ... SDG mapping
- completed_at ................ ISO-8601 UTC timestamp
- quantity, unit .............. Impact measurement
- location, latitude, longitude ... Geographic data
- proof_url ................... Evidence reference
- evidence_post_ids ........... Related post IDs
- difficulty, points .......... Challenge metadata

Why This Works:
1. Deterministic: Same data always produces same hash (no randomness)
2. Independent: Verification doesn't need Module 2 - can work offline
3. Immutable: Once on blockchain, hash is permanent timestamp
4. Transparent: Anyone can verify the integrity
